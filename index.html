Digital ITAD Internal Management Software

This project is designed to provide inventory management services for Digital ITAD.

This web application will include the ability to

  • View all items, boxes, and pallets in inventory, both shipped and on-hand
  • View all boxes in a pallet, items in a box, and item details
  • View all purchase orders, receiving orders, and invoices
  • Create and edit purchase orders and invoices
  • Edit receiving orders
  • Generate pick tickets for invoices
  • Mark invoices as shipped
  • View, create, and edit customers and suppliers
  • Generate reports for box and pallet labels
  • Generate reports for pick tickets, packing lists, and invoices

The web application will have a companion android software to be used with scanners

Notable Libraries

  • MaterialUI - used for custom UI components, like versatile data grids and input text fields
  • React-pdf - used to generate PDF docs for reports
  • jsdoc - used to generate documentation for the site
    • To generate docs, run jsdoc -c .\.jsdoc.conf.json -R README.md in the terminal from the root directory
    • To access docs, go to docs folder and find index.html. Right click and either open with a browser (like Edge) or with an extension (like Live Server)
    • To just view the docs without generation, visit this link

File Structure:

  • assets - Images, global styles, and other non-code-related files
  • components - Global components for the project (i.e. not page-specific). Organized into folders based on functionality/component type
  • context - Custom contexts for the project. Contexts are used to reduce code redundancy, as well as to raise state to a global level. Some contexts that included are a loading overlay and the snackbar alert box.
  • data - All static data used in the project. This includes routing links and elements, sidebar links, and homepage sections.
  • hooks - All non-context custom hooks for the project, such as accessing API endpoints.
  • pages - Pages for the project, including the main "App" page that routes everything together. Each page has its own folder, with the page's component being the index.js in that folder. All page-specific components and styles will be put in that folder as well. Generally, if a component is shared between pages, it will be in the 'components' folder and not a page folder.
  • utils - All pure functions for the project. A pure function is one that doesn't require outside dynamic data (i.e. data from the 'data' directory is okay), doesn't have any side effects (ex. sorting an array in-place), and will return the same output given the same input (counter example: random number generator).

ITAD Workflow:

Digital ITAD has a very specific way in which they work. Here is the current workflow as we understand it, which will most likely be updated as we refine this product:

  • A Purchase Order is created with specific information
    • The user can choose whether the Purchase Order is an Interco order or not
    • If it’s an Interco order, no contact information, as well as Purchase Order Items are entered
    • If it’s not an Interco order, contact information is needed, and Purchase Order Items in the form of Description, Quantity, and Price Per Unit.
  • Once a Purchase Order is created, it can be edited up to the point of receiving. Once an Order has received an item, it can no longer be edited.
  • A Purchase Order can be reviewed to see the original order and what was asked for, as well as what was received. This shows the discrepancies between the two and the real amount to be paid.
  • Items are received from a shipment and scanned in by the Android Scanner. It takes care of creating Item records and marking the Purchase Order as received.
  • Items received from a shipment are placed directly into a Box, into a Tray, or onto a Pallet.
    • Only Desktops are placed onto Pallets
    • Only RAM sticks and CPUs are placed into Trays
    • Everything else is placed into a Box, grouped by type/generation/version/like-characteristics
  • Tray Workflow:
    • A Tray consists of Tray Items, which are any number of like RAM sticks and CPUs.
    • A Tray is created at the same time as the Tray Items, where they enter the specific information about the bundle of RAM sticks or number of CPUs
    • RAM sticks generally will come in bundles of 25 or 50, making that the general size of a Tray with RAM
    • CPUs will go one-by-one normally, though the functionality exists for them to be groups together in the database. Therefore, the user can choose to put the CPU in the next open CPU Tray, which will have a maximum capacity of 21.
  • Once Items are placed into Containers (Boxes/Pallets/Trays), they can be moved to any location (or shelf in the case of Trays) that they need to via the Android Scanner. They move wherever until they are shipped out.
  • An Invoice is created by entering in Supplier, Customer, & Contact information, as well as whatever Boxes and Pallets to be shipped in the Invoice. These Boxes and Pallets are only on-hand (not shipped) and need to be not already assigned to an Invoice.
  • Once an Invoice is created, it can be viewed by the Items to be shipped out (Invoice) and the Containers to be picked (Order Picking).
  • Once an Invoice is completed and shipped, it can be marked as completed, which marks all Containers and Items to be shipped.
